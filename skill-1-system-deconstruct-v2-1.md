# Skill 1: System Deconstruct (v2 — Source-Agnostic)
**Position in pipeline:** Runs AFTER Skill 0 GREENLIGHT. Consumes the locked winning package; produces the locked system matrix that Skill 2 scripts from.
**Trigger Phrase:** `Skill 1: Deconstruct [Source]`

`[Source]` may be: a book, a paper or study, or a documented real-world system (e.g. a production method, an institutional process, a historical mechanism). State explicitly which source class is in use and whether deconstruction draws on supplied text or canonical knowledge. If canonical knowledge, flag any low-confidence areas for the Skill 2.5 factual claims register.

---

## PURPOSE

Convert the greenlit concept's source material into a dense, drawable system matrix. The matrix, not the source, is what gets scripted. Everything downstream (Skill 2 staging, Skill 3 manifests) consumes modules from this matrix, so every module must map cleanly onto one of the five layout architectures.

---

## EXECUTION PROTOCOL

1. **Strip.** Remove all casual filler, motivational fluff, repetitive anecdotes, and biographical padding from the source. Anecdotes survive only if they are load-bearing evidence for a mechanism, and they are compressed to their causal skeleton.

2. **Isolate.** Extract the core structural frameworks, operational mental models, or behavioural feedback loops. Each becomes a candidate module.

3. **Reframe.** Express every module in precise, systemic language — inputs, triggers, bottleneck states, gain coefficients, compounding loops, failure modes — never generic self-help phrasing.

4. **Map.** Tag every module with its natural layout architecture:
   - Comparative states / portfolios → **3-Column Bento Grid**
   - Cause-effect machinery, paired forces → **Split-Screen Mechanical Node Diagram**
   - Sequences, decay, compounding over time → **Linear Vector Timeline**
   - Feedback loops, habit cycles, reinforcement → **Central Interlocking Loop**
   - Calculations, live variables, equations resolving → **Live Terminal Code Matrix**
   A module that maps to none is rewritten until it does, or cut.

5. **Order.** Sequence modules into logical phases that serve the locked package's promise: the phase structure must build toward the package's open loop closing in the final third.

---

## OUTPUT FORMAT

A technical Markdown system matrix:

1. **Engine statement** — the whole system in one sentence of cold mechanics.
2. **Module table** — for each module: name, mechanism description (systemic language), inputs/outputs, failure mode, mapped layout architecture, and the accent-colour role of its key element (cyan signal, amber limiting step, red defect).
3. **Phase sequence** — the ordered narrative phases, each listing its modules and its function in the loop structure (open, escalate, foreshadow, close).
4. **Source ledger** — source class, what was drawn from supplied text vs canonical knowledge, and any claims pre-flagged NEEDS CHECK.

---

## HARD RULES
- One locked matrix per greenlit package. No parallel versions.
- The matrix serves the locked package's promise; modules that don't serve it are cut regardless of how interesting they are in the source.
- No module ships without a layout architecture mapping.
- The source is credited in the ledger, not in the framing. The matrix describes a system, not a book.
