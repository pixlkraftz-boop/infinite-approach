# Skill 2: Technical Screenplay
**Position in pipeline:** Runs AFTER Skill 1. Consumes the locked system matrix; produces the two-column production script that Skill 2.5 audits. No script reaches Skill 2.5 without a locked matrix behind it.
**Trigger Phrase:** `Skill 2: Screenplay [Phase / Full Script]`

---

## PURPOSE

Convert the locked system matrix into a render-ready two-column production script: a single-narrator UK voiceover track engineered for retention, paired scene-by-scene with deterministic visual staging instructions. The script is written to be audited mechanically by Skill 2.5, so every law below is treated as a hard spec, not a style preference.

---

## INPUTS

1. The locked winning package from Skill 0 (title, thumbnail concept, core promise).
2. The locked system matrix from Skill 1 (modules, phase sequence, layout mappings, accent roles, source ledger).
3. Declared runtime target. Word budget = runtime minutes × one hundred forty five words.

If any input is missing, stop and request it. Do not script from the source directly; the matrix is the only scripting source.

---

## EXECUTION PROTOCOL

### 1. Scene Plan
Before writing, lay out the scene table: scene number, timestamp allocation, matrix module(s) consumed, loop function (open / escalate / foreshadow / close), and layout architecture inherited from the module mapping. The cold open loop must close in the final third, foreshadowed at roughly one third and two thirds.

### 2. Audio Column (Left)
Narrative-driven UK voiceover obeying every law:

**Register.** Measured, objective, deeply analytical. BBC Horizon. Cold operational mechanics, never self-help phrasing, never YouTube hype vocabulary. Flowing natural spoken English; no choppy stylised fragments. Vary sentence length for rhythm.

**Hook.** First forty words of Scene 01 must contain a concrete open loop: an unanswered question, a withheld identity, or a stake. No greetings, no channel references, no statement of intent to explain.

**Retention laws.**
- Rehook at least every two hundred twenty words: a new loop, tension injection, or pattern interrupt.
- But/Therefore density above eighty percent. Never connect beats with chronological and-then transitions.
- Every paragraph delivers at least one progression element: new information, progression toward goal, obstacle, or emotional change.
- No loop closes without a new one opening within ten seconds of audio, except the final payoff.
- Golden ratio: roughly thirty seconds of context per sixty seconds of mechanism in action.

**Audio Protocol (TTS safety).** Forbidden in spoken text: dashes of any kind, colons, semicolons, parentheses, brackets, braces, double quotation marks, digits, and the symbols percent ampersand plus equals slash backslash hash at asterisk underscore tilde caret angle brackets. All numbers, years, and figures written fully as spoken words. Permitted: period, comma, apostrophe, question mark. Exclamation marks and ellipses are off-register; avoid. Never reference the edit; narration stands entirely on its own.

### 3. Visual Column (Right)
For every scene specify:
- **Layout architecture** — exactly one of the five (3-Column Bento Grid, Split-Screen Mechanical Node Diagram, Linear Vector Timeline, Central Interlocking Loop, Live Terminal Code Matrix) or a declared phase variant. Multi-phase scenes are split into separate visual phases, each naming its own architecture.
- **Asset geometry** — the vector structures, blocks, paths, and grids to render, in the channel registry style (canvas #0A0E12, faint dot grid, #E8ECEF strokes at 1.25px monoline, monospace schematic type only).
- **Accent placement** — where cyan #27C8D9 (signal), amber #E8A33D (limiting step), red #D9442B (defect) appear, consistent with the module's accent role from the matrix.
- **Schematic whitelist** — the complete list of strings permitted to render in that scene (node labels, axis variables, units, mathematical glyphs, terminal readouts). Anything outside the whitelist is a Visual Law breach.
- **Sequential build behaviour** — the order in which elements draw, slide, morph, or fade, keyed loosely to narration beats (exact word-cue binding happens in Skill 3).

**Visual Law.** No post-production text of any kind: no titles, subtitles, captions, bullet slides. If it looks like an editor typed it, it is banned. Recurring channel motifs (amber limiting node, coefficient bars, dot-grid origin point) are deployed where the matrix calls for them.

---

## OUTPUT FORMAT

1. Scene plan table.
2. The two-column script, scene-marked with timestamps at approximately one hundred forty five words per minute.
3. **Compliance notes:**
   - Total word count vs runtime target, and per-scene counts vs allocation.
   - Loop map: every loop, where opened, where foreshadowed, where closed.
   - Factual claims list pre-flagged for the Skill 2.5 register, carrying forward every NEEDS CHECK item from the Skill 1 source ledger.

---

## HARD RULES
- One script per locked matrix. No parallel drafts past this gate.
- The script may compress or reorder matrix modules, but may not introduce mechanisms absent from the matrix.
- Self-compliance notes do not substitute for the Skill 2.5 audit. The script is not cleared until Skill 2.5 says RENDER CLEARED.
